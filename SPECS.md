# 浮世塔罗 Specs

## 1. 项目概述

浮世塔罗是一个面向中文用户的浏览器端塔罗体验应用。它的核心定位不是“随机抽牌玩具”，而是一个可以完成提问、抽牌、解读、归档、分享和回看闭环的单页产品。

产品形态：

- 单页应用
- 纯前端
- 本地存储优先
- 中文内容优先

## 2. 产品目标

### 2.1 主要目标

- 提供完整的塔罗占卜路径，而不是只展示一张结果牌
- 支持从轻量日签到深度牌阵的不同使用场景
- 让每次解读都可以被沉淀为可检索、可比较、可分享的记录
- 保持纯前端部署和使用成本，避免依赖后端服务

### 2.2 非目标

- 不做账号体系
- 不做云同步
- 不做服务端推荐或 AI 生成内容
- 不做支付、订阅、后台管理
- 不做多页内容站或 CMS

## 3. 目标用户与使用场景

### 3.1 目标用户

- 习惯中文语境的塔罗初学者
- 需要轻量日常指引的用户
- 会重复查看、比较和归档解读记录的重度用户

### 3.2 核心场景

- 快速查看今天最值得留意的能量
- 围绕一个具体问题完成一次完整抽牌和解读
- 将结果收藏、添加追问、后续复盘
- 导出文案或海报分享到聊天、社媒或个人笔记
- 在牌卡百科中延伸阅读本次抽到的牌

## 4. 信息架构

应用采用长滚动单页结构，核心区块如下：

1. Hero
2. 每日指引
3. 引导面板
4. 占卜工作台
5. 结果解读
6. 记录中心
7. 牌卡百科

导航方式：

- 页面内锚点滚动
- 当前区块通过 `IntersectionObserver` 更新导航高亮
- 不使用前端路由

懒加载边界：

- `RecordCenter`
- `CardEncyclopedia`

这两个区块采用 `React.lazy` + 视口进入时再渲染的双重延迟策略。

## 5. 功能规格

### 5.1 每日一张

目标：

- 为当天提供一个低门槛入口

行为：

- 基于本地日期生成固定 seed
- 使用固定问题“今天最值得留意的能量是什么？”
- 使用 `daily-energy` 单张牌阵
- 支持揭晓、分享文案和下载海报
- 揭晓后写入 `kind = daily` 的记录

约束：

- 同一天的记录 id 固定
- 揭晓状态为当前会话内状态，刷新后会重置

### 5.2 占卜工作台

输入项：

- 占卜问题
- 主题
- 牌阵
- 牌阵模式（如该牌阵提供变体）

行为：

- 提供按主题切换的问题建议
- 只有问题、主题和牌阵都具备时才允许抽牌
- 同一问题组合在 30 秒内重抽时，需要二次确认

### 5.3 牌阵系统

当前牌阵数：11

已知牌阵：

- 单张指引
- 每日一张
- 圣三角
- 抉择罗盘
- 关系映照
- 七日流向
- 五张月度牌阵
- 四季牌阵
- 凯尔特十字
- 黄道十二宫
- 生命之树

要求：

- 每个牌阵必须定义位置、提示语、布局类型和卡数
- 部分位置可限定抽牌池，如仅大阿尔卡那或特定元素
- 变体牌阵可覆盖默认位置定义

### 5.4 抽牌与解读

抽牌规则：

- 单次解读内不允许重复抽到同一张牌
- 支持正位 / 逆位
- 支持 `random` 与 `up-only` 两种朝向策略

结果生成：

- 生成逐位置解读
- 生成整体总结
- 生成建议列表
- 生成行动计划
- 生成气质标签与主导信号

### 5.5 结果解读

必须支持：

- 逐张揭晓
- 全部揭晓
- 记录标题编辑
- 记录标签编辑
- 行动计划勾选
- 追问记录
- 收藏
- 分享文案
- 海报导出

自动归档规则：

- 普通占卜在全部位置都揭晓后自动归档
- `全部揭晓` 直接触发归档

### 5.6 记录中心

记录类型：

- `reading`
- `daily`

支持能力：

- 查看全部记录
- 查看收藏记录
- 查看自动归档记录
- 查看每日记录
- 文本搜索
- 标签筛选
- 日期范围筛选
- JSON 导出
- JSON 导入
- 最多两条记录对比

### 5.7 牌卡百科

支持能力：

- 默认聚焦本次抽到的牌
- 切换到完整 78 张牌库
- 查看牌义、建议和扩展说明
- 支持搜索与分组浏览

### 5.8 分享与导出

分享策略：

- 优先调用 `navigator.share`
- 退化到剪贴板复制

海报导出：

- 使用 SVG 模板生成视觉内容
- 绘制到 canvas
- 输出 PNG 下载

## 6. 技术架构

### 6.1 技术栈

- Vite 7
- React 19
- TypeScript 5
- ESLint 9
- Vitest 4
- Playwright 1.58
- IndexedDB (`idb`)
- `sharp`

### 6.2 目录责任

- `src/App.tsx`: 主装配与区块编排
- `src/sections`: 主功能区块
- `src/components`: 通用与复合组件
- `src/data`: 静态业务数据
- `src/domain`: 类型模型
- `src/engine`: 核心业务引擎
- `src/hooks`: 会话与持久化 hook
- `src/lib`: 辅助工具
- `src/test`: 单元与组件测试
- `tests/e2e`: 浏览器流测试

### 6.3 关键模块

#### `useReadingSession`

负责普通占卜会话状态，包括：

- 输入态
- 抽牌态
- 揭晓态
- 记录标题与标签
- 分享状态
- 追问状态
- 自动归档和收藏行为

#### `createReading`

负责：

- 解析牌阵
- 抽卡
- 构造牌面视图
- 生成解读、总结、建议和行动计划

#### `usePersistentRecords`

负责：

- 启动时读取记录
- 写入记录
- 导入导出
- 与 IndexedDB / localStorage 协调

## 7. 数据模型

### 7.1 核心运行时模型

- `ReadingInput`
- `ReadingResult`
- `ReadingCardView`
- `PositionReading`

### 7.2 持久化模型

统一使用 `ReadingRecordV2`，字段包含：

- 记录类型
- 收藏状态
- 标题与问题
- 主题与牌阵元数据
- 卡牌列表
- 总结与信号
- 标签
- 行动计划完成状态
- 追问
- 每日反思结构

### 7.3 id 规则

- 普通占卜记录 id 由问题、主题、牌阵、变体、抽到的牌和朝向共同计算
- 每日记录 id 基于本地日期生成

这意味着：

- 相同普通占卜结果会合并，而不是无限新增重复记录
- 同一天的每日记录会覆盖更新到同一条数据

## 8. 存储与迁移

### 8.1 当前存储策略

- `localStorage` 作为同步启动入口
- IndexedDB 作为更稳定的持久化层
- 记录默认按 `updatedAt` 排序
- 记录上限为 120 条

### 8.2 兼容与迁移

当 `ukiyo-tarot.records-v2` 不存在时，系统会尝试迁移：

- `ukiyo-tarot.saved-readings`
- `ukiyo-tarot:reading-history`

### 8.3 导入导出

- 导出为 JSON 包装对象
- 导入支持数组或 `{ records }` 包装格式
- 导入后与现有记录按合并规则融合

## 10. 已知约束与风险

### 10.1 已知约束

- 无账号与云同步
- 无服务端存储
- 无多浏览器同步
- 资源衍生图生成需手动执行，不自动并入构建

### 10.2 已知风险

- IndexedDB 启动时若已有数据，会优先回填，当前未对比其与 localStorage 的新旧时间
- 每日记录的本地日历语义与 UTC 时间戳存在潜在跨日偏差
- 每日揭晓状态不持久化，刷新页面后会重新回到未揭晓态
- 导入 JSON 的结构校验偏浅，重点只校验顶层形状
- `buildRecordShareText()` 已存在，但当前文档上未确认是否有已保存记录的直接分享入口
